##  Section 01. 소프트웨어 개발 프로세스의 이해

### 1. 프로세스(Process)의 정의
* **개념:** 일이 처리되는 과정이나 공정, 또는 주어진 일을 해결하기 위한 목적으로 순서가 정해져 수행되는 일련의 절차
* **소프트웨어 개발 프로세스:** 소프트웨어를 개발할 때 작업 순서의 틀을 제공하여 개발 생산성과 품질을 높이고, 정해진 기간과 예산 내에 고품질의 제품을 납품할 수 있도록 돕는 절차

### 2. 소프트웨어 프로세스 모델 (Software Process Model)
* **정의:** 소프트웨어 개발 생명주기(SDLC)를 체계적으로 관리하기 위해 개발 활동의 순서, 작업 내용, 산출물 등을 정형화한 동적 프레임워크

##  Section 02. 전통적 소프트웨어 프로세스 모델 종류 및 특징

### 1. 주먹구구식 개발 모델 (Build and Fix Model)

* **특징:** 공식적인 계획, 분석, 설계 단계 없이 일단 코딩을 한 후(Build), 수정하는 방식(Fix)을 반복
* **단점:** 정해진 개발 문서가 없어 유지보수가 매우 어렵고, 프로젝트의 진행 상황을 파악할 수 없음. 나중에 아키텍처를 수정하려면 비용이 기하급수적으로 증가

### 2. 폭포수 모델 (Waterfall Model / 선형 순차적 모델)

* **개념:** 요구분석, 설계, 구현, 테스트, 유지보수가 **하늘에서 물이 떨어지듯 순차적으로 진행**되는 전통적인 모델
* **특징:**
* 각 단계가 끝나면 확실한 산출물(문서)이 나와야 다음 단계 진행
* 적용 경험이 많아 기술적 위험이 낮고 관리가 수월


* **단점 :**
* **요구사항 변경이 불가능에 가까움:** 후반부 테스트 단계에서 결함이 발견되면 초기 단계로 돌아가기가 매우 어려움
* 사용자가 최종 제품을 볼 수 있는 시점이 개발 프로세스의 맨 마지막(구현 및 테스트 이후)이라 피드백이 늦음



### 3. V 모델 (V-Model / 검증 및 확인 중심)

* **개념:** 폭포수 모델의 변형, 개발 단계와 **테스트(검증/확인) 단계를 매칭하여 'V자' 구조**로 시각화한 모델
* **특징:** 테스팅 활동을 개발 초기(요구분석, 설계)부터 함께 계획 -> 결함을 조기에 발견할 수 있어 품질 보증에 강함

### 4. 프로토타입 모델 (Prototyping Model / 진화적 프로세스)

* **개념:** 사용자의 요구사항을 바탕으로 실제 작동하는 모형(시제품/Prototype)을 빠르게 제작하여 사용자에게 먼저 보여주는 모델
* **장점:** 요구사항을 명확히 파악할 수 있고, 사용자 만족도가 높으며 오해를 조기에 방지
* **단점:** 시제품 제작에 따른 추가 비용 및 일정 산정의 어려움, 사용자가 시제품을 완성품으로 오해할 수 있음

### 5.나선형 모델 (Spiral Model)

* **개념:** 보엠(Boehm)이 제안한 모델로, 시스템을 여러 번 반복 개발하면서 점진적으로 완성해 나가는 방식
* **4대 주요 활동 순서:**
1. **목표 설정 (Planning):** 목적, 제약 조건, 대안 결정
2. **위험 분석 (Risk Analysis):** 프로젝트 실패 요인을 식별하고 **위험을 최소화**하기 위한 전략 수립
3. **개발 및 검증 (Engineering):** 소프트웨어 개발 및 테스트 진행
4. **고객 평가 및 다음 단계 계획 (Evaluation):** 결과물 평가 후 다음 주기 계획


* **장점:** **위험 분석 단계**가 있어 대규모, 고비용, 복잡한 시스템 개발 시 프로젝트 실패 확률을 낮춤

### 6. 통합 프로세스 모델 (UP: Unified Process)

* **개념:** 객체지향 분석·설계(UML)를 위한 대표적인 반복적·점진적 개발 프로세스 모델
* **4대 개발 단계 :**
1. **도입 (Inception):** 프로젝트 비즈니스 모델 정립, 범위 설정, 타당성 조사
2. **구체화 (Elaboration):** 아키텍처 설계 완료, 핵심 위험 식별 및 해결, 세부 계획 수립
3. **구축 (Construction):** 실제 시스템을 컴포넌트 기반으로 대부분 구현 및 테스트 완성
4. **전이 (Transition):** 완성된 소프트웨어를 사용자 환경으로 인도, 베타 테스트, 교육 수행


* **공통 작업 영역:** 각 단계마다 비즈니스 모델링, 요구사항 정의, 분석 및 설계, 구현, 테스트, 배치, 형상 관리, 프로젝트 관리가 **반복적(Iteration)으로 공통 수행**됨

##  Section 03. 현대적 모델: 애자일 프로세스 모델 (Agile)

### 1. 애자일(Agile)의 정의

* **의미:** '날렵한', '민첩한'이라는 뜻으로, 변화를 거부하기보다 **변화에 민첩하고 유연하게 대응**하여 가치 있는 소프트웨어를 빠르게 제공하는 방법론
* **배경:** 계획과 문서 위주의 무거운 전통적 방법론(폭포수 등)의 한계를 극복하기 위해 등장

### 2.애자일 선언문 (4대 기본 가치)


* **① 프로세스와 도구보다 ➡️ 개인과 상호작용**
* **② 광범위한 문서보다 ➡️ 작동하는 소프트웨어**
* **③ 계약 협상보다 ➡️ 고객과의 협력**
* **④ 계획을 따르기보다 ➡️ 변화에 대응하는 것**
